# State Artifact Composition

PJM source builders emit source-specific artifacts plus optional view-ready
context fields. Morning State Packs still accept one immutable artifact mapping,
so multi-source direct-PJM runs need deterministic composition before candidate
pack creation.

`compose_artifact_payloads` merges artifact payloads with explicit rules:

- text summaries are joined in input order;
- drivers, deltas, evidence, scenarios, source lineage, and known series lists
  are concatenated;
- `inputs`, market scope, and view sections are deep-merged;
- source-specific state products remain separate keys; and
- ambiguous key collisions fail closed.

The composed output includes `artifact_composition` metadata with payload count,
composition product keys, view fields, input keys, current-day view keys, and
shared list counts. Input payloads may not supply this reserved key; it is
generated by the composer so downstream validation can audit what was assembled.

State-pack validation checks this metadata when it is present. If a composed
artifact payload is edited after composition and the metadata no longer matches
the actual artifact mapping, candidate validation fails before publish.

Composition product keys may include source products and approved derived
products. They are not equivalent to source-product keys.

This keeps load fundamentals, generation mix, hourly price products, and derived
price shapes physically separate while allowing `HotState` analyst views to see
one composed read payload.

Composition does not publish state, mutate accepted packs, create synthetic
values, or approve new source semantics. It is a pre-publish assembly step used
before the existing candidate-build and atomic-publish path.
